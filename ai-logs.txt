Recommendation
Use a relational DB (Postgres + Prisma), and generate graph JSON (nodes, edges) from relational data for the UI.
This is the fastest credible path with your stack, easiest to explain, and best aligned with “NL → SQL dynamically.”

Tradeoffs

Relational DB (recommended)
Pros: Prisma-native, SQL generation is direct, easy migrations, easier README/story, strong for tabular business data.
Cons: Deep multi-hop traversals are less elegant than graph DBs.
Graph DB
Pros: Native traversal semantics, graph-first querying.
Cons: Adds stack complexity, weaker fit for “translate to SQL,” more to justify in take-home.
Call: For this assignment, graph DB is not clearly justified unless requirements are traversal-heavy (many variable-depth hops).
Architecture overview

Ingestion layer
Parse CSV/JSON dataset.
Map into normalized entities + relationships.
Upsert via Prisma.
Data model
Domain tables (e.g., Company, Person, Deal, etc.).
Relationship table(s) or explicit join tables.
Graph service
Reads relational tables and emits UI-ready graph JSON.
NL query service pipeline
Relevance gate: classify if prompt is in-domain; reject unrelated.
SQL generation: LLM gets strict schema context and outputs SQL only.
SQL validator: allowlist tables/columns, SELECT-only, add LIMIT, block risky patterns.
Execution: run validated SQL.
Answer synthesis: produce response strictly from returned rows; if no rows, say insufficient data.
UI
Single-page app: graph canvas + query box + results panel.
Tailwind for layout, Framer Motion for light transitions only.
